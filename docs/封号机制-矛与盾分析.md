# Claude 封号机制:矛与盾分析

> 目的:在研究账号池反检测前,先把 Anthropic 的检测分层(盾)拆清楚,再把当前代码(矛)逐层对照,定位漏点。

---

## 一、Anthropic 的"盾" — 风控分层

把它想成 **四层漏斗**,任一层不自洽都会进灰名单(429 / 降权 / 封号)。

### L1 网络层指纹(传输层)

- **TLS ClientHello**:JA3/JA4 = (cipher list, extension list+order, supported_groups, signature_algorithms, ALPN, key_share group)。真 CC 用 Bun→BoringSSL,**目标 JA3 = `d871d02cecbde59abbf8f4806134addf`**。axios/reqwest/curl 各有自己的指纹,一眼能区分。
- **HTTP/2 指纹**:SETTINGS 帧(INITIAL_WINDOW_SIZE / MAX_FRAME_SIZE …)、WINDOW_UPDATE、HEADERS 伪头序(`:method :path :scheme :authority` 排列)、PRIORITY 帧。可以独立打出 **Akamai HTTP/2 fingerprint**。
- **HTTP/1.1 头序+大小写**:Go/Python 会把头规范化,Node undici 保持原大小写且**有固定顺序**:
  ```
  accept → content-type → user-agent → x-claude-code-session-id
  → x-stainless-* → anthropic-* → x-app
  → (传输头由库追加在末尾)
  ```
- **加密内容指纹**:`accept-encoding` 值序(真 CC 是 `gzip, deflate, br, zstd`)、有没有 `Connection: keep-alive` 这种 HTTP/1 残留。

### L2 客户端身份层(自报家门)

五项必须 **互洽**:

1. **UA**:`claude-cli/2.1.156 (external, cli)`
2. **X-Stainless-Package-Version**:`0.94.0` ↔ 必须匹配 UA 那个 CC 版本带的 SDK
3. **X-Stainless-OS / Arch / Runtime / Runtime-Version**:`MacOS / arm64 / node / v24.3.0` 形成"一台真实机器"
4. **anthropic-beta**:beta flag 集合是版本相关的白名单 — 旧版用不上新 flag,新版不会缺
5. **x-app / anthropic-version / anthropic-dangerous-direct-browser-access**:固定常量,缺一个就是非 CC

> 关键自洽:UA 的 OS 不能和 X-Stainless-OS 互相矛盾,UA 版本不能和 anthropic-beta 跨代。

### L3 请求体语义层(深度自检)

- **`metadata.user_id`**:CC 2.1.x 是 `{"device_id","account_uuid","session_id"}` JSON 字符串。
  - `device_id` 必须 64 位 hex
  - `session_id` 必须 UUID v4
  - 多出 `email/org` 字段 → 老版本/伪造
- **`cc_version=X.Y.Z.<3hex>; cch=<5hex>`** —— 写在 system block 第一行 `x-anthropic-billing-header:` 里:
  - `3hex` = SHA256(salt + 末条 user 消息位置 [4,7,20] 字节 + 版本) 取前 3 位
  - `5hex` = **xxhash64(整个序列化 body, seed=0x6E52736AC806831E) & 0xFFFFF**
  - 这是 **整 body 的密封戳**,任何字段改动都必须重算。服务端会复算比对。
- **system prompt banner**:首句必须 `"You are Claude Code, Anthropic's official CLI for Claude."`,后面跟 OS/Platform/Shell/Working directory/Git user/system-reminder 块。banner 内的 Platform/OS/Shell **必须**和 X-Stainless-OS 同一台机器。
- **字段白名单**:CC 不会发 `temperature / top_p / top_k / stop_sequences / tool_choice`,带了就是 API SDK 露馅。`tools` 必须是数组(可空),`stream:true`,`max_tokens ≤ 32768`。
- **缓存断点**:真 CC 把 billing 行独占在 `system[0]` 且 **不带** `cache_control`,断点打在后面静态块上。位置乱了 → 缓存命中率异常 → 风控信号。

### L4 遥测自洽层(自己出卖自己)

- **`/v1/event_logging/batch`**:CC 周期上报 `device_id / email / account_uuid / organization_uuid / env / process`,**这些值必须和 OAuth token claims 对得上**。代理转 token 但不改遥测 → token 里 sub=A,遥测说 device=B → 立刻露馅。
- **`/api/eval/{clientKey}` (GrowthBook 远程评估)**:attributes 里有 `deviceID / email / accountUUID / organizationUUID / subscriptionType / platform / appVersion / apiBaseUrlHost` —— **`apiBaseUrlHost` 是杀手字段**,它会暴露代理地址。
- **`process` 字段**(base64 编码 JSON):`rss / heapTotal / heapUsed / constrainedMemory`,真实 Node 进程数据。如果是常量或不动 → 一眼假。

### L5 行为层(账号画像)

- **单账号同时多 session**:8 个 device_id 同时活跃 = 共号实锤
- **跨大区切换速度**:5 分钟内 US→SG IP 是不可能的人类行为
- **prompt 缓存命中率**:真人会反复触发同一 system 段,命中率正常应该高;每次都 miss = 像在拼装请求
- **5 小时窗口配额**:订阅号每 5h 用量,超阈值后调用 1M context 会 429 `"Usage credits are required"` —— 这本身是计费信号但会被记录
- **OAuth refresh 频率**:正常 CC 每 1h 左右 refresh 一次,过快/过慢都是异常

## 三、综合判断 — 当前最大的三个漏点

按风险从高到低:

### 🔴 1. HTTP/2 指纹是裸奔(L1 盲区)

JA3 做到顶级,但 reqwest 的 h2 SETTINGS 和 undici 不同。Anthropic 完全可以用 **Akamai HTTP/2 fingerprint**(`fp.h2.fingerprint`)单独识别,你的 TLS 完美也没用。

**建议**:抓包对比 reqwest 和真 CC 的 SETTINGS 帧、HEADERS 伪头序;或者切到 h3 / curl-impersonate 风格的 h2 patch。

### 🔴 2. sticky 跨小时切换的"device_id 漂移"(L5 行为层)

`session_hash` 含 `hour_window`,跨小时同一会话会重新选号 → 新账号 → 新 device_id,而 **会话内容里的 working_dir / virtual_project 由 session_id 派生是稳定的**。后端看到 "同一台机器(虚拟项目稳定)突然换了 device_id" → 共号信号。

**建议**:要么去掉 hour_window 让 session 直到 TTL 都粘账号,要么 session_id 也随账号变。

### 🟡 3. banner 太短 + tools 字段未归一化(L3 语义)

- 注入模式的 banner 只有一行,真 CC banner 后面跟着大段 system-reminder
- `normalize_cc_identity` 没覆盖 `tools[].description`,MCP 工具说明里的路径/用户名会泄漏

### 次要(已经做对,但值得 review)

- `derive_account_uuid` 没打 UUID v4 版本位 —— 严格校验会 fail
- `BUN_CIPHER` 没有 GREASE —— 比对真抓包确认
- token refresh 节奏没看到主动错峰
