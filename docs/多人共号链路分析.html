多人共号 · 链路与防封漏点分析

一个 Anthropic 账号反代给多人用,从请求进来 → 客户端模拟/归一化 → 转发到官方的完整链路,以及暴露多人的漏点。

my-ccb gateway · 基于代码精读(rewriter.rs / gateway.rs / client_guard.rs)+ 历史使用数据

核心判断:单请求伪装"接近真",但账号整体行为"不像一个人"。

  • 单个请求层面:TLS / HTTP2 指纹、header 顺序大小写、身份字段都已统一成真 CC —— 很接近,但不是 100%:存在「版本 ↔ 系统提示词不一致」和「cch 校验戳不重算」两个内容缺口,混版本多人时会露馅。
  • 账号整体层面:一个号 = 几十人的体量 / 并发 / 时间画像,这是任何单请求伪装都藏不住的。
  • 所以「单人本地永远没事、一反代多人就死」= 混版本触发内容不一致 + 聚合行为暴露多人 两个原因叠加。

3漏点 —— 按严重度排

高 · 确凿

A. 版本不一致(header 改了,body 没改)

归一化只改了 header 里的版本(UA / package / runtime → 账号吸取的版本),但 body 里 system 提示词的版本相关内容没改:真 CC 系统提示里带 cc_version=2.1.119.xxx 版本 token,不同版本措辞/工具 schema 也不同。混版本时 header 说一个版本、body 是另一个版本的内容。

用户A 跑 v2.1.119 →
　header: UA = claude-cli/2.1.160（被改成账号吸取版本）
　body  : cc_version = 2.1.119（原样未改）
→ 官方一比对:"header 说 160,提示词却是 119" = 篡改/异常
高 · 疑似

B. cch 校验戳没重算

真 CC 在 system 里带 cch=xxxxx 校验戳,由(首条用户消息几个字节 + 版本)算出。代码里 API 注入模式会重算 cch,normalize 模式完全没碰:我们 scrub 改了内容、改了 header 版本,cch 还是客户端用原内容+原版本算的旧值。若官方服务端重算比对 → 对不上 = 篡改信号。作者在 API 模式特意处理、normalize 模式漏掉,是明确的不对称缺口。

中

C. 部分 header 没归一化

normalize_os_headers_ordered 只改了 7 个 header,其余原样透传。其中 anthropic-version 不同 CC 版本可能不同 → 没改 = 漏;metadata.user_id 里若有 device_id/account_uuid 之外的字段(email/org)也不改 = 漏。

根本 · 改不掉

D. 行为 / 聚合特征(为什么单人绝对安全、多人必死)

就算每个请求都伪装完美,一个账号的整体行为画像仍然不是人。这是统计行为,不在单个请求里,任何 per-request 伪装都藏不住。RPM 削峰 + 会话上限是在压这一类,只能缓解、不能消除。

· 体量:一个号承载几十人的请求量
· 并发:同时 20+ 个不同 session(没人会开 24 个并行 CC 对话)
· 时间:24h 不间断,没有人类睡觉/吃饭空档
· 多样性:同一刻在几十个项目/语言/CLAUDE.md 上并行干活 —— 人是串行的

4能补的(按性价比)

补 A · 推荐先做

版本一致化

normalize 时把 body 系统提示里的 cc_version= 等版本 token 一起改成账号版本;或更稳:按客户端自己的真实版本归一化,不强行统一(版本本就允许不同用户不同),更不易穿帮。

补 B

cch 重算

normalize 模式也走一遍 cch 重算,复用 API 模式那段逻辑,让校验戳和最终内容一致。

补 C

补齐 header

把 anthropic-version 也纳入归一化;user_id 里多余字段一并清掉。

压 D · 进行中

压聚合特征

已上线 RPM 削峰 + 会话上限;核心是多上号、把每号压到像单人(实测推荐 18/号 RPM、5 并发会话)。
