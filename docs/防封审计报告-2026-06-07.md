# 防封系统审计报告

> 审计日期:2026-06-07
> 审计范围:`src/tlsfp/`、`src/service/rewriter.rs`、`src/service/account.rs`、`src/service/probe.rs`、`src/model/identity.rs`、`src/service/gateway.rs`
> 结论:当前不封号是因为 80% 常见信号已灭,但仍有若干**高危项**等着上游升级风控时被触发。

## 一、当前已经做了什么(防封覆盖面)

### 1. TLS 指纹层
`src/tlsfp/tlsfp.rs` + `craftls/`(私改 rustls)

- 伪装成真实 Claude Code 使用的 **Bun / BoringSSL**,JA3 目标 `d871d02cecbde59abbf8f4806134addf`
- **17 个 cipher**、**14 个 extension(顺序固定不打乱)**、**ALPN 强制 http/1.1**
- 支持 **X25519+ML-KEM-768 后量子混合密钥交换**(Bun 真有的)

### 2. 请求改写层
`src/service/rewriter.rs`(1667 行)

- **Header 白名单 + 大小写还原**:CC 模式只放行真客户端会发的 header,AI Gateway / CloudFlare / 自定义遥测 header 全部剥离
- **anthropic-beta 对齐**:补齐 `cache-diagnosis-2026-04-07`、`interleaved-thinking-2025-05-14` 等真客户端会带的 beta
- **Billing 块隔离**(commit `2a19f32`):`x-anthropic-billing-header` 单独放一个 system block,**不加 cache_control**,避免击穿 prompt 缓存
- **cch attestation**(commit `99f65fb`):对 normalize 后的 body 做 xxhash64,按 billing_mode 门控
- **System prompt 脱敏**:平台、OS、工作目录、git 身份全部按账号归一
- **遥测体改写**(`/event_logging/batch`):device_id / email / account_uuid / process / GrowthBook 全部归一

### 3. 虚拟身份层
`src/model/identity.rs`

- 每账号一次性生成稳定身份:`device_id`、Node 版本、平台预设、CC 版本、内存档
- **多人共号归一化**:`SHA256(account_email)` → 从 24 个稳定 (username, git_name) 对里固定挑一个

### 4. 账号调度层
`src/service/account.rs`

- **Sticky session**:24h 内同会话锁定同账号
- **并发槽**:Redis 计数,满了排队 20s
- **429 智能隔离**:解析 `anthropic-ratelimit-unified-{5h,7d}-*` 头,分级隔离

### 5. 健康探活层
`src/service/probe.rs`(未提交)

- `X-Probe-Check: 1` 触发短路,不走 rewriter、不占账号槽、不消耗配额

## 二、缺点 / 风险(按严重程度排序)

### 🔴 高危(真会被识别)

#### H1. TLS 指纹单版本硬编码,且 GREASE 不打乱
**位置**:`src/tlsfp/tlsfp.rs:139-248`、`tlsfp.rs:247`

- 17 个 cipher + 14 个 extension 都是 `static`,锁死在某一版 Bun 上,**Bun 升级后即过期**
- `shuffle_extensions: false` —— 真 BoringSSL 每次握手 GREASE 位置会换,你每次 ClientHello **字节级别完全相同**
- 上游记录第一次握手的扩展序列,后续所有请求都能关联到同一"机器"

#### H2. ALPN 强制 `http/1.1`
**位置**:`src/tlsfp/tlsfp.rs:194`

- 真现代 Claude Code 用 h2 的话,h1 降级是**二进制信号**
- 没做 h2 支持是单点死穴

#### H3. 身份池只有 24 对(username, git_name)
**位置**:`src/model/identity.rs:106-115`

- 1000 个账号撞 24 个槽 → **平均每个槽 ~42 个账号共用同一 git 身份**
- 上游做账号聚类时会非常显眼:"这 42 个账号都是 `john / John Smith`"

#### H4. CCH 哈希依赖 JSON canonicalization,但没真做 canonicalize
**位置**:`src/service/rewriter.rs:1132-1145`

- xxhash64 算的是 `serde_json::to_vec` 输出
- serde_json 对 object 的 key 顺序保留输入顺序 → **不同来源的 body 算出来的 cch 会漂移**
- 一漂移就击穿 prompt 缓存,等于白做

#### H5. 出口 IP 完全不轮换
**位置**:`src/service/gateway.rs:653-666`

- 每账号一个 reqwest client,绑死一个 proxy_url
- 多账号同一台机器出口 → "account_123 永远来自 1.2.3.4"
- **这是目前单一最大风险**,比 TLS 指纹还容易关联,真用户不可能这样

### 🟡 中危(放大后会暴露)

#### M1. Sticky session 哈希跨用户撞车
**位置**:`src/service/account.rs:840-909`

- `SHA256(UA + 首条消息 + 小时窗)` —— 两个用户同一小时跑 `claude /init`,会绑到同一账号 24h
- 第一个用户烧完配额被隔离 → 第二个用户瞬间换号 → 上游看到"同一会话突然换设备"

#### M2. Node 版本只有 3 个预设
**位置**:`src/model/identity.rs:8-59`

- 真实用户分布是连续的(v20~v24 一堆小版本)
- 1000 个账号只有 3 种 Node 版本 → **离散聚类一眼看出**

#### M3. 内存档只有 3 档(16/32/64GB)
**位置**:`src/model/identity.rs:84-88`

- 真机内存连续分布,你三档 → 1000 个账号约 333/333/333 的完美三峰
- heap 抖动救不了 RSS 这种粗粒度信号

#### M4. 没有任何请求时序抖动
**位置**:`src/service/gateway.rs`(无时序逻辑)

- 收到请求 → ~10ms 转发出去
- 真人打字 / 等思考 → 秒级停顿
- 行为指纹,做不做 jitter 差距巨大

#### M5. OAuth refresh 机械化定时
**位置**:`src/service/account.rs:19`(OAUTH_REFRESH_BUFFER_SECONDS = 5 * 60)

- 固定提前 5 分钟刷,所有账号都这个规律
- 真人是用的时候触发刷新 → 时间分布是请求驱动的,不是过期驱动的

---

### 🟢 低危(可以慢慢改)

#### L1. device_id 用 `thread_rng` + 格式不对
**位置**:`src/model/identity.rs:90-95`

- 非 CSPRNG,理论上有 preimage 风险
- **真 CC 用的是 UUID v4 格式**,你是 64 字符 hex —— 格式本身就不对

#### L2. tools 块完全不归一化
**位置**:`src/service/rewriter.rs`(无 tools 处理)

- system prompt 改得很细,但 `tools` 数组原样转发
- 用户自定义工具描述会原汁原味漏出去,变成跨请求稳定指纹

#### L3. API 模式下用户自带 system prompt 不脱敏
**位置**:`src/service/rewriter.rs:285-344`

- API 模式只往前面 prepend CC banner,**用户自己的 prompt 原文上去**
- 含家目录、git config、shell 历史的话直接泄露

#### L4. 多处 `.unwrap()` 在请求热路径
**位置**:`src/service/rewriter.rs:336/376/411/450/463/479/890/898/1051-1059`

- 客户端发奇怪 body 能让 worker panic,error log 会带账号 ID + 请求体

#### L5. Probe 响应可被反向探测
**位置**:`src/service/probe.rs:73-117`

- 固定 `msg_probe` ID + 固定 1/1 tokens
- 上游探一下就知道这是个网关

## 三、修复优先级

如果只能修 3 个,按收益排序:

| 优先级 | 修复项 | 收益 | 工作量 |
|--------|--------|------|--------|
| **P0** | **出口 IP 轮换**(H5) | 灭掉目前最显眼的关联信号 | 中(加 SOCKS5 池或每账号独立住宅 IP) |
| **P0** | **GREASE 随机化 + Bun 版本探活**(H1、H2) | 灭掉 TLS 层的"字节级相同"信号 | 大(craftls 改造 + CI 抓包对齐) |
| **P1** | **扩大身份池到 200+ + Node 版本连续化**(H3、M2、M3) | 灭掉离散聚类信号 | 小(改预设表) |

剩下的 jitter、CCH canonicalize、tools 归一化是**积累伤害**,慢慢补就行,短期不致命。

## 四、为什么现在不封号

封号本质是**行为指纹聚合识别**,Anthropic 检测共号/代理的信号大致是:

| 信号 | 当前防御 | 状态 |
|------|----------|------|
| TLS 握手不像 Bun | TLS 层伪装 | ✅ 灭(但单版本) |
| UA / beta / x-stainless 不对 | Header 改写 | ✅ 灭 |
| device_id / OS / home / git 跳变 | 身份归一 | ✅ 灭(但池太小) |
| cch 对不上 body | xxhash attestation | ⚠️ 半灭(canonicalize 缺失) |
| billing 行污染缓存 | block 隔离 | ✅ 灭 |
| 会话跨账号跳 | sticky session | ✅ 灭(但有撞车) |
| 多用户并发打满 | 并发槽 + 429 隔离 | ✅ 灭 |
| **多账号同 IP 出口** | **无** | ❌ **未灭** |
| **TLS GREASE 字节级相同** | **无** | ❌ **未灭** |
| **请求时序无人类停顿** | **无** | ❌ **未灭** |

**结论**:每一层单独并不够,叠起来才把上游看到的"多用户共号"伪装成"单台机器单人使用"。这是它现在稳定的原因。

但是 **H5(IP 不轮换)是定时炸弹** —— Anthropic 一旦在风控里加入"按账号-IP 关联度"维度,大量账号会瞬间一起死。**这个必须先修**。
